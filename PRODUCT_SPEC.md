## 1. 서비스 개요 및 Admin 포지셔닝

### 1-1. 서비스 비전 및 Admin의 역할

**Happiness**는 사진 작가(photographer)가 포트폴리오를 업로드하고, 촬영을 원하는 일반 사용자가 작가에게 직접 촬영을 의뢰할 수 있는 C2C 크리에이터 마켓플레이스다.

**Admin 대시보드의 핵심 역할:**

- **플랫폼 신뢰 보호**: 부적절 콘텐츠 모더레이션, 신고 처리, 스팸 문의 차단을 통해 사용자 경험 보호
- **성장 촉진**: 추천 콘텐츠 큐레이션, 작가 인증 관리로 우수 크리에이터 육성
- **데이터 기반 운영**: 문의 성사율, 작가 퍼포먼스, 콘텐츠 트렌드를 수치로 파악하고 정책에 반영
- **서비스 안정화**: 시스템 점검 모드, Rate Limit 설정, 관리자 활동 감사(audit)로 운영 리스크 최소화

### 1-2. 사용자 유형 및 권한 체계 (WM/SA 업무 분장)

현재 시스템에는 `Authority` 열거형으로 3가지 역할이 정의되어 있다.

| 역할 코드 | 명칭 | 설명 |
|-----------|------|------|
| `WM` | 웹관리자 | 최고 권한. 시스템 설정, 관리자 계정 생성, 모든 데이터 수정/삭제 가능 |
| `SA` | 운영자 | 콘텐츠 모더레이션, 회원 상태 관리, 공지 작성 등 일상 운영 업무 담당 |
| `US` | 일반 사용자 | 앱 사용자. Admin 접근 불가 |

**WM (웹관리자) 전용 업무:**

- 관리자 계정 생성 및 권한 변경
- 시스템 환경 설정 (Rate Limit, 이메일 서버, 점검 모드)
- 콘텐츠 정렬 정책 / 노출 가중치 설정
- 관리자 활동 로그 조회 (전체)
- 수익화 지표 및 고급 리포트 접근

**SA (운영자) 담당 업무:**

- 회원 상태 변경 (정지/해제), 부적절 콘텐츠 삭제
- 신고 건 검토 및 처리
- 공지사항/배너 등록 및 수정
- 작가 인증 신청 검토
- 문의 이상 탐지 처리
- 통계 조회 및 리포트 내보내기

**정책 제안:** WM와 SA 권한을 엔티티 레벨에서 Spring Security `@PreAuthorize`로 구분하여, SA가 WM 전용 기능에 접근할 경우 403 Forbidden을 반환하도록 한다. 현재 `ROLE_WM`, `ROLE_SA` 모두 `/api/admin/**`에 접근 가능하나, 민감 엔드포인트는 `ROLE_WM`만 허용하도록 세분화가 필요하다.

## 2. 현황 분석 및 GAP 분석

### 2-1. 현재 Admin 기능 현황

| 도메인 | 구현된 기능 | 미비 사항 |
|--------|-------------|-----------|
| 회원 관리 | 목록/검색, 역할 변경(WM/SA/US), 삭제 | 상세 페이지 없음, 상태 관리(정지/해제) UI 없음, 벌크 액션 없음 |

| 문의 관리 | 목록 조회, 읽음/미읽음 필터, 촬영종류 필터, 읽음 처리, 삭제 | 성사율 통계 없음, 스팸 탐지 없음 |
| 시리즈 관리 | 목록 조회, 삭제 | 추천 시리즈 설정 없음 |
| 통계 | 일별 트렌드, 무드 분포 파이차트, 촬영종류 바차트, 인기사진 TOP10 | 코호트 분석 없음, 작가별 퍼포먼스 없음, 리포트 내보내기 없음 |
| 시스템 | 이메일 설정 조회, Rate Limit 조회, DB 상태 조회 | 실시간 수정 불가, 점검 모드 없음, 관리자 활동 로그 없음 |
| 공지/배너 | 미구현 | 전체 미구현 |
| 모더레이션 | 미구현 | 신고 시스템, 자동 플래그, 제재 기록 전체 미구현 |
| 작가 인증 | 미구현 | Verified 뱃지, 신청 워크플로우 전체 미구현 |
| 태그/카테고리 | 미구현 (하드코딩) | shootType, colorMood 모두 코드에 고정값으로 존재 |

### 2-2. 실무 운영에서 누락된 기능 목록 (우선순위 포함)

| 우선순위 | 기능 | 비즈니스 임팩트 | 근거 |
|----------|------|----------------|------|
| P0 (즉시) | 회원 상태 관리 UI (정지/해제) | 매우 높음 | `MemberStatus.SUSPENDED`는 엔티티에 존재하나 Admin UI가 없어 운영 불가 |
| P0 (즉시) | 회원 상세 페이지 | 높음 | 특정 회원 민원 처리 시 해당 회원의 사진/문의/활동을 한 화면에서 볼 수 없음 |
| P0 (즉시) | 신고 관리 시스템 | 매우 높음 | 사용자 신고를 받을 채널이 없어 플랫폼 신뢰도 위협 |
| P1 (단기) | 공지사항/배너 관리 | 높음 | 서비스 점검, 이벤트 공지를 앱에 노출할 방법이 없음 |
| P1 (단기) | 작가 인증 워크플로우 | 높음 | 검증된 작가를 구분할 수 없어 사용자 신뢰 저하 |
| P1 (단기) | 추천/피처드 콘텐츠 편집 | 중간 | 메인 노출 사진이 알고리즘에만 의존하여 큐레이션 불가 |
| P2 (중기) | 문의 성사율 통계 | 중간 | 핵심 비즈니스 지표인 문의→촬영 전환율을 측정할 수 없음 |
| P2 (중기) | 작가 퍼포먼스 대시보드 | 중간 | 우수 작가 육성 정책 수립을 위한 데이터 부재 |
| P2 (중기) | shootType/colorMood 관리 | 중간 | 신규 카테고리 추가 시 코드 배포가 필요한 구조 |
| P3 (장기) | 코호트 분석 | 낮음 | 성장 단계에서 필요한 리텐션 지표 |
| P3 (장기) | 수익화 지표 KPI | 낮음 | 유료화 전환 전 미리 설계 필요 |

### 2-3. 앱(클라이언트) ↔ Admin 불일치 현황

| 항목 | 앱(클라이언트) 동작 | Admin 현황 | 불일치 리스크 |
|------|-------------------|------------|--------------|
| 사진 정렬 기준 | 앱에서 "인기순/최신순" 탭으로 클라이언트 결정 | Admin에서 기본 정렬 정책 설정 불가 | 앱 기본 노출 순서를 운영팀이 제어할 수 없음 |
| colorMood 값 | 앱에서 WARM/COOL/NEUTRAL/VIVID/DARK/SOFT 6가지 사용 | Admin 필터는 동일 6가지 (정합) | 신규 무드 추가 시 코드 배포 필요 |
| shootType 값 | 웨딩/가족/프로필/스냅/바디프로필/커플 6가지 | Admin 필터는 동일 6가지 (정합) | 신규 종류 추가 시 코드 배포 필요 |
| MemberStatus | 앱에서 SUSPENDED 회원은 로그인 차단 (추정) | Admin에서 상태 변경 UI 없음 | 정지 처리를 DB 직접 조작해야 함 |
| 피처드 콘텐츠 | 앱 메인에 피처드 섹션이 있다면 Admin과 연동 필요 | Admin에 피처드 관리 없음 | 메인 노출 콘텐츠를 코드로만 제어 |
| 작가 verified 뱃지 | 앱에서 verified 작가 표시 (추정) | Admin에 인증 관리 없음 | 인증 부여/회수 불가 |

## 3. 콘텐츠 정책 관리 기능 기획

### 3-1. 정렬 정책 관리

**목적:** 앱의 사진 피드 기본 정렬 순서를 Admin에서 설정하여, 신규 작가 노출 기회 조정, 이벤트 기간 특정 무드 우선 노출 등 운영 정책을 코드 배포 없이 실현한다.

**운영팀이 할 수 있는 것:**
- 앱 메인 피드의 기본 정렬 기준을 "최신순 → 인기순"으로 즉시 전환
- 특정 기간에 "저장순" 가중치를 높여 저장 수 기반 알고리즘 노출 촉진
- 신규 작가의 사진에 일정 기간 "신규 작가 부스트" 가중치 적용

**화면 구성:**

```
[정렬 정책 관리] 페이지
┌──────────────────────────────────────────────────────┐
│  앱 기본 정렬 기준                                   │
│  ◉ 최신순 (createdAt DESC)                           │
│  ○ 인기순 (likesCount DESC)                          │
│  ○ 저장순 (savesCount DESC)                          │
│  ○ 공유순 (sharesCount DESC)                         │
│  ○ 복합 점수순 (가중치 직접 설정)                    │
│                                                      │
│  복합 점수 가중치 (합계 = 100%)                      │
│  좋아요: [35]%  저장: [40]%  공유: [25]%             │
│                                                      │
│  신규 작가 부스트 (가입 후 N일 이내)                 │
│  활성화: [ON]  대상: 가입 후 [30]일 이내             │
│  부스트 배수: [1.5]배                                │
│                                                      │
│  [미리보기]  [저장]  마지막 수정: 2026-06-15 14:32   │
└──────────────────────────────────────────────────────┘
```

**입력 항목:**

| 항목 | 타입 | 유효성 |
|------|------|--------|
| `defaultSortBy` | enum (LATEST/LIKES/SAVES/SHARES/COMPOSITE) | 필수 |
| `likesWeight` | integer (0~100) | 복합 선택 시 필수, 합계 100 |
| `savesWeight` | integer (0~100) | 복합 선택 시 필수 |
| `sharesWeight` | integer (0~100) | 복합 선택 시 필수 |
| `newAuthorBoostEnabled` | boolean | 선택 |
| `newAuthorBoostDays` | integer (1~365) | 부스트 활성 시 필수 |
| `newAuthorBoostMultiplier` | decimal (1.0~5.0) | 부스트 활성 시 필수 |

**정책:**
- 설정값은 DB의 `policy_settings` 테이블에 key-value 형태로 저장
- 앱 API 서버는 매 요청 시 캐시(TTL 5분)에서 정책 조회
- 복합 점수 가중치 합계가 100이 아닐 경우 저장 불가

**엣지 케이스:**
- 복합 점수 가중치 합계 != 100: "가중치 합계는 100%여야 합니다" 오류 표시
- 부스트 배수를 1.0 미만으로 설정 시도: 1.0 미만 불허 처리
- 정책 변경 즉시 적용 여부: 저장 시 캐시 무효화 트리거, 최대 5분 내 반영

### 3-2. 추천/피처드 콘텐츠 편집

**목적:** 앱 메인 화면의 "에디터 픽", "이번 주 추천 작가" 섹션에 노출할 사진 및 시리즈를 운영팀이 직접 선정하여 큐레이션 품질을 높인다.

**운영팀이 할 수 있는 것:**
- 주간 에디터 픽 사진 10장 선정 및 순서 지정
- 이번 달 추천 시리즈 5개 선정
- 피처드 기간(시작일/종료일) 설정으로 자동 노출/종료

**화면 구성:**

```
[피처드 콘텐츠 관리] 페이지
┌─────────────────────────────────────────────────┐
│  탭: [에디터 픽 사진] [추천 시리즈] [추천 작가] │
└─────────────────────────────────────────────────┘

[에디터 픽 사진 탭]
┌──────────────────────────────────────────────────┐
│  현재 피처드 사진 (7개 / 최대 10개)              │
│  ┌────┐ ┌────┐ ┌────┐                           │
│  │[1] │ │[2] │ │[3] │  ... ← 드래그로 순서 변경 │
│  └────┘ └────┘ └────┘                           │
│  노출 기간: 2026-06-15 ~ 2026-06-21              │
│  [사진 추가]  [기간 수정]  [전체 초기화]         │
└──────────────────────────────────────────────────┘

[사진 추가 모달]
- 사진 ID 직접 입력 or 사진 검색 (제목/작가명)
- 검색 결과 썸네일 미리보기
- 이미 피처드된 사진 선택 시 경고
```

**입력 항목:**

| 항목 | 설명 |
|------|------|
| `featuredPhotoIds` | 피처드 사진 ID 목록 (순서 포함) |
| `featuredSeriesIds` | 피처드 시리즈 ID 목록 |
| `startDate` | 노출 시작일시 |
| `endDate` | 노출 종료일시 |
| `memo` | 내부 운영 메모 (앱 미노출) |

**정책:**
- 에디터 픽 최대 10개. 10개 초과 시 추가 불가.
- 종료일이 지난 피처드는 앱에서 자동으로 일반 알고리즘 정렬로 대체
- 피처드로 선정된 사진이 삭제될 경우: 피처드 목록에서 자동 제거, 슬롯 공백 유지(재편집 필요 알림)

**엣지 케이스:**
- 피처드 설정 중 해당 사진이 신고로 숨김 처리될 경우: 자동 피처드 해제 + SA에게 이메일 알림
- 시작일 > 종료일 설정 시도: "종료일은 시작일 이후여야 합니다" 유효성 오류
- 동일 사진을 에디터 픽과 추천 시리즈 커버에 중복 사용: 허용 (별도 슬롯)

### 3-3. 컬러무드별 노출 가중치 설정

**목적:** 특정 시즌/이벤트에 맞게 무드별 노출 비중을 조절한다. 예: 봄 시즌에는 WARM/SOFT 무드 사진을 더 많이 노출, 할로윈 기간에는 DARK 무드 부스트.

**화면 구성:**

```
[컬러무드 가중치 설정]
┌──────────────────────────────────────────────┐
│  무드       기본 가중치   현재 적용 가중치   │
│  WARM       1.0x         [1.5]x  ← 수정 가능│
│  COOL       1.0x         [1.0]x             │
│  NEUTRAL    1.0x         [1.0]x             │
│  VIVID      1.0x         [0.8]x             │
│  DARK       1.0x         [1.0]x             │
│  SOFT       1.0x         [1.3]x             │
│                                             │
│  적용 기간: [2026-06-01] ~ [2026-06-30]     │
│  메모: 6월 봄/여름 시즌 WARM/SOFT 부스트    │
│                                             │
│  [기본값으로 초기화]  [저장]                │
└──────────────────────────────────────────────┘
```

**정책:**
- 가중치 범위: 0.1x ~ 5.0x
- 기간이 종료되면 자동으로 1.0x(기본값) 복귀
- 가중치 변경 이력은 운영 로그에 기록

**엣지 케이스:**
- 모든 무드를 0x로 설정하려 할 경우: 최소 1개 무드는 0x 이상이어야 하므로 불허
- 기간 중복 설정: 기간이 겹치는 가중치 설정은 최신 설정이 우선 적용

## 4. 회원 관리 고도화 기획

### 4-1. 회원 상세 페이지 (드릴다운)

**목적:** 특정 회원에 대한 민원 처리, 제재 검토 시 해당 회원의 모든 활동을 한 화면에서 파악하여 신속하게 의사 결정할 수 있도록 한다.

**운영팀이 할 수 있는 것:**
- 회원 기본 정보 확인 (이름, 이메일, 가입일, 소셜 로그인 여부, 상태)
- 해당 회원이 업로드한 사진 목록 확인 및 개별 삭제
- 해당 회원이 수신한 문의 목록 확인
- 해당 회원이 발송한 문의 목록 확인
- 해당 회원이 작성한 시리즈 목록 확인
- 이 회원에게 적용된 제재 이력 확인
- 상태 변경(정지/해제/탈퇴 처리) 즉시 실행

**화면 구성:**

```
[회원 상세] /admin/members/:id
┌─────────────────────────────────────────────────────┐
│ [← 목록으로]  회원 상세: 김민준 (@user1)            │
│                                                     │
│ ┌──────────────────┐  ┌──────────────────────────┐  │
│ │ 기본 정보        │  │ 상태 관리                │  │
│ │ 이메일: user1@.. │  │ 현재: ACTIVE [정지하기]  │  │
│ │ 역할: US         │  │ 누적 신고: 2건           │  │
│ │ 가입: 2026-03-01 │  │ 제재 이력: 1건           │  │
│ │ 소셜: local      │  └──────────────────────────┘  │
│ └──────────────────┘                               │
│                                                     │
│  탭: [사진(12)] [받은 문의(5)] [보낸 문의(8)]       │
│       [시리즈(3)] [제재 이력(1)]                    │
│                                                     │
│  [사진 탭]                                          │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐                       │
│  │    │ │    │ │    │ │    │  ← 그리드, 각 카드에  │
│  │[삭]│ │[삭]│ │[삭]│ │[삭]│     개별 삭제 버튼    │
│  └────┘ └────┘ └────┘ └────┘                       │
└─────────────────────────────────────────────────────┘
```

**탭별 표시 항목:**

| 탭 | 표시 항목 |
|----|-----------|
| 사진 | 썸네일, 제목, 무드, 업로드일, 좋아요/저장/공유 수, 삭제 버튼 |
| 받은 문의 | 발신자, 촬영종류, 촬영희망일, 예산, 접수일, 읽음 여부 |
| 보낸 문의 | 수신자, 촬영종류, 촬영희망일, 예산, 접수일 |
| 시리즈 | 커버, 제목, 사진 수, 생성일, 삭제 버튼 |
| 제재 이력 | 제재 유형, 사유, 기간, 처리자(Admin), 처리일 |

**엣지 케이스:**
- 사진 0장인 신규 회원: 사진 탭 "등록된 사진이 없습니다" 빈 상태 표시
- 탈퇴(DELETED) 회원 접근: 개인정보 마스킹 처리 후 열람 가능 (이름 "탈퇴회원", 이메일 "***@***.***")

### 4-2. 작가 인증(Verified) 관리 워크플로우

**목적:** 플랫폼 내 전문 작가를 검증하여 Verified 뱃지를 부여함으로써 사용자 신뢰도를 높이고, 인증 작가의 작품이 우선 노출될 수 있는 기반을 마련한다.

**운영팀이 할 수 있는 것:**
- 인증 신청 목록 검토
- 신청서 상세 확인 (포트폴리오 링크, 활동 경력, SNS 주소)
- 인증 승인 / 반려 (반려 사유 입력 필수)
- 인증된 작가 목록 관리 및 인증 취소

**워크플로우:**

```
[앱에서 작가 인증 신청]
       ↓
[Admin: 인증 신청 목록 - 상태: PENDING]
       ↓
[SA/WM 검토]
   ↓           ↓
[승인]       [반려]
   ↓           ↓
[Member.isVerified = true]  [반려 사유 이메일 발송]
[앱에 Verified 뱃지 표시]
```

**필요한 DB 변경:**

```
Member 엔티티에 추가 필요:
- isVerified: boolean (default false)
- verifiedAt: LocalDateTime

VerificationRequest 신규 엔티티:
- id, member, status(PENDING/APPROVED/REJECTED)
- portfolioUrl, careerDescription, snsUrl
- adminMemo, processedBy(Member), processedAt
- createdAt
```

**화면 구성:**

```
[작가 인증 관리] /admin/verifications
┌───────────────────────────────────────────────┐
│ 탭: [신청 대기(3)] [승인됨(12)] [반려됨(5)]   │
│                                               │
│ [신청 대기 탭]                                │
│ 신청자  | 신청일      | 포트폴리오 | 상태  | 처리 │
│ 이서연  | 2026-06-15  | [링크]     | 대기  | [승인][반려] │
│ 박지호  | 2026-06-14  | [링크]     | 대기  | [승인][반려] │
│                                               │
│ [승인 클릭 시]                                │
│  → "이서연(@user2)의 작가 인증을 승인합니다." │
│    내부 메모: [              ]                │
│    [확인]  [취소]                             │
│                                               │
│ [반려 클릭 시]                                │
│  → 반려 사유 (필수): [               ]        │
│    [반려 처리]  [취소]                        │
└───────────────────────────────────────────────┘
```

**정책:**
- 반려 사유는 신청자 이메일로 자동 발송 (이메일 설정이 완료된 경우)
- 인증 취소 시에도 취소 사유를 기록하고 작가에게 통보
- 동일 회원의 재신청은 반려 후 30일 이후 가능

**엣지 케이스:**
- 이미 인증된 작가가 재신청한 경우: "이미 인증된 작가입니다" 오류 반환
- 정지(SUSPENDED) 상태 회원의 인증 신청: 신청 접수 차단
- 인증 후 작가가 탈퇴한 경우: `isVerified` 자동으로 false 처리

### 4-3. 회원 상태 관리 (정지/해제/탈퇴)

**목적:** 플랫폼 규정 위반 회원을 신속하게 정지 처리하고, 정지 기간/사유를 기록하여 이의 신청 시 근거 자료로 활용한다.

**현재 `MemberStatus` 열거형:** `ACTIVE`, `INACTIVE`, `SUSPENDED`, `DELETED`

**상태 전환 규칙:**

```
ACTIVE ──[정지]──→ SUSPENDED ──[해제]──→ ACTIVE
ACTIVE ──[비활성화]──→ INACTIVE ──[재활성화]──→ ACTIVE
ACTIVE / SUSPENDED ──[탈퇴 처리]──→ DELETED
DELETED → (전환 불가, 읽기 전용)
```

**정지 처리 입력 항목:**

| 항목 | 타입 | 필수 여부 |
|------|------|----------|
| `suspendReason` | 텍스트 (최대 500자) | 필수 |
| `suspendUntil` | 날짜 (무기한 선택 가능) | 선택 |
| `notifyUser` | boolean | 선택 (이메일 발송 여부) |
| `internalNote` | 텍스트 | 선택 |

**필요한 DB 변경:**

```
Member 엔티티에 추가 필요:
- suspendReason: String
- suspendUntil: LocalDateTime (null = 무기한)
- suspendedBy: Long (처리한 Admin ID)
- suspendedAt: LocalDateTime
```

**엣지 케이스:**
- WM 계정(Authority.WM)은 정지 불가: "웹관리자 계정은 정지할 수 없습니다" 오류
- 정지 기간이 만료된 경우: 스케줄러가 매일 새벽 2시에 `SUSPENDED → ACTIVE` 자동 전환
- 이미 DELETED 상태인 회원에 정지 시도: "탈퇴된 회원입니다" 오류
- 정지 회원이 앱 로그인 시도: Spring Security 에서 `UserDetails.isEnabled()` false 반환으로 차단

### 4-4. 회원 벌크 액션

**목적:** 다수의 스팸 계정, 비활성 계정을 한번에 처리하여 운영 효율성을 높인다.

**운영팀이 할 수 있는 것:**
- 목록에서 체크박스로 다수 회원 선택
- 선택한 회원 전체에 동일한 역할 변경 적용
- 선택한 회원 전체 정지 처리 (동일 사유 적용)
- 선택한 비활성 회원 일괄 정리(DELETED 전환)

**화면 구성:**

```
[회원 목록]
□ 전체선택  | 이름 | 이메일 | 역할 | 상태 | 가입일 | 관리
☑ 전체선택  ← 1명 이상 선택 시 액션 바 표시

[선택 시 액션 바]
┌──────────────────────────────────────────────┐
│ 3명 선택됨  [역할 변경 ▼]  [정지]  [삭제]  │
└──────────────────────────────────────────────┘
```

**정책:**
- 벌크 삭제는 WM 전용. SA는 벌크 정지까지만 가능.
- 벌크 처리 대상에 WM 계정이 포함된 경우 해당 계정은 자동으로 처리 대상에서 제외되고 경고 표시
- 처리 건수가 50건 초과 시 비동기 처리 + 완료 알림 제공

**엣지 케이스:**
- 벌크 삭제 시도 중 일부 회원에게 진행 중인 문의가 있는 경우: "N명에게 진행 중인 문의가 있습니다. 계속하시겠습니까?" 추가 확인
- 처리 도중 네트워크 오류 발생: 처리 완료된 건/실패한 건을 분리하여 결과 리포트 표시

## 5. 콘텐츠 모더레이션 기획

### 5-1. 신고 관리 시스템

**목적:** 사용자가 부적절한 사진/회원을 신고할 수 있는 채널을 만들고, 운영팀이 신고를 검토하여 신속하게 처리함으로써 플랫폼 신뢰도를 보호한다.

**워크플로우:**

```
[앱 사용자: 신고 접수]
       ↓
[신고 접수 DB 저장 (status: PENDING)]
       ↓
[Admin: 신고 목록 조회]
       ↓
[SA/WM 검토]
   ↓                    ↓
[처리 결정]           [기각]
  ↓                    ↓
[콘텐츠 숨김/삭제]   [기각 완료로 상태 변경]
[신고자 처리 결과 통보]
```

**필요한 신규 엔티티:**

```java
Report {
    id: Long
    reporter: Member          // 신고자
    targetType: String        // PHOTO / MEMBER / SERIES
    targetId: Long            // 신고 대상 ID
    reportType: String        // INAPPROPRIATE / SPAM / COPYRIGHT / ABUSE / OTHER
    description: String (TEXT) // 신고 내용 상세
    status: String            // PENDING / REVIEWING / ACTION_TAKEN / DISMISSED
    adminNote: String         // 처리자 메모
    processedBy: Member       // 처리한 Admin
    processedAt: LocalDateTime
    createdAt: LocalDateTime
}
```

**화면 구성:**

```
[신고 관리] /admin/reports
┌───────────────────────────────────────────────────────┐
│  요약 카드                                            │
│  [대기 중: 12건]  [검토 중: 3건]  [처리 완료: 89건]  │
│                                                       │
│  필터: [전체] [사진] [회원] [시리즈]                  │
│        [전체 상태] [대기중] [검토중]                  │
│                                                       │
│  신고자  | 대상 유형 | 대상    | 신고 유형  | 접수일   | 상태  | 처리 │
│  이서연  | 사진      | #23     | 부적절     | 06-15   | 대기  | [검토] │
│  박지호  | 회원      | @user5  | 스팸       | 06-14   | 검토중| [처리▼] │
└───────────────────────────────────────────────────────┘
```

**신고 상세/처리 화면:**

```
[신고 상세]
- 신고자: 이서연 (@user2), 신고일: 2026-06-15
- 대상: [사진 #23 보기] "봄의 기억" by @user1
- 신고 유형: 부적절 콘텐츠
- 신고 내용: "노출 수위가 과도합니다..."

[처리 액션]
○ 콘텐츠 숨김 (앱에서 비노출, 복구 가능)
○ 콘텐츠 삭제 (영구 삭제)
○ 작성자 경고
○ 작성자 정지
○ 기각 (신고 내용 부적절)

처리자 메모: [                                    ]
[처리 완료]  [취소]
```

**신고 유형 정의:**

| 코드 | 표시명 | 설명 |
|------|--------|------|
| `INAPPROPRIATE` | 부적절 콘텐츠 | 노출/혐오 등 |
| `SPAM` | 스팸/광고 | 반복 도배, 광고성 |
| `COPYRIGHT` | 저작권 침해 | 타인 사진 무단 도용 |
| `ABUSE` | 괴롭힘/욕설 | 특정인 공격 |
| `OTHER` | 기타 | 상세 입력 필수 |

**정책:**
- 동일 콘텐츠에 신고 3건 이상 누적 시 자동으로 SA에게 슬랙/이메일 알림 발송
- 처리 완료 후 신고자에게 처리 결과 이메일 발송 (이메일 설정 완료 시)
- 기각 처리된 신고도 이력으로 보존 (삭제 불가)

**엣지 케이스:**
- 이미 삭제된 사진에 대한 신고: "해당 콘텐츠가 이미 삭제되었습니다"로 자동 처리(DISMISSED)
- 동일 사용자가 동일 콘텐츠를 중복 신고: 접수 차단, "이미 신고하신 콘텐츠입니다" 앱 안내
- 신고 처리 도중 대상 회원이 탈퇴: 신고 이력은 보존, 대상 표시는 "탈퇴회원"으로 변경

### 5-2. 부적절 콘텐츠 자동 플래그 + 수동 검토 큐

**목적:** 모든 신고를 일일이 수동 처리하는 부담을 줄이기 위해, 규칙 기반 자동 플래그 시스템을 도입하여 명백히 문제가 있는 콘텐츠를 우선 검토 큐에 올린다.

**자동 플래그 트리거 규칙:**

| 규칙 | 임계값 | 액션 |
|------|--------|------|
| 단기간 신고 집중 | 24시간 내 동일 콘텐츠 신고 3건 이상 | 검토 큐 자동 등록 |

| 반복 신고 유저 | 동일 회원이 7일 내 10건 이상 신고 수신 | 고위험 플래그 |

## 6. 문의(Inquiry) 관리 고도화

### 6-1. 문의 성사율 통계 (접수→연결→촬영 완료 퍼널)

**목적:** "문의가 실제 촬영 계약으로 이어지는 비율"을 측정하여 플랫폼 핵심 지표를 파악하고, 성사율이 낮은 원인을 분석하여 개선한다.

**현재 한계:** `Inquiry` 엔티티에는 `isRead` 필드만 있어 "작가가 응답했는지", "촬영이 성사되었는지" 추적 불가.

**단계적 접근:**

**단기 (현재 엔티티로 측정 가능한 지표):**

| 지표 | 계산 방법 |
|------|-----------|
| 총 문의 접수 | `COUNT(inquiries)` |
| 읽음률 | `COUNT(isRead=true) / COUNT(*)` |
| 응답 지연 문의 | 접수 후 48시간 이상 미읽음 건수 |
| 촬영종류별 문의 수 | `GROUP BY shootType` |

**중기 (엔티티 확장 후 측정):**

`Inquiry` 엔티티에 상태 필드 추가:

```java
// InquiryStatus 추가 제안
enum InquiryStatus {
    PENDING,      // 접수, 작가 미읽음
    READ,         // 작가 읽음
    RESPONDED,    // 작가 응답 (앱에서 응답 기능 추가 시)
    CONTRACTED,   // 계약 완료 (작가가 직접 마킹)
    CANCELLED     // 취소/거절
}
```

**화면 구성:**

```
[문의 성사율 통계] /admin/stats/inquiry-funnel
┌────────────────────────────────────────────────┐
│  기간 선택: [이번달 ▼]  [커스텀: 시작~종료]   │
│                                                │
│  퍼널 차트                                     │
│  ████████████████  접수: 145건 (100%)          │
│  ████████████       읽음: 118건 (81%)          │
│  ████████           응답: 97건 (67%)           │
│  ████               계약: 43건 (30%)           │
│                                                │
│  촬영종류별 성사율                             │
│  웨딩: 45%  가족: 28%  프로필: 38% ...        │

```

### 6-2. 촬영종류별 수요-공급 갭 분석

**목적:** 특정 촬영종류에 문의가 집중되는데 해당 종류의 작가가 적은 경우를 파악하여, 작가 모집 마케팅의 방향성을 결정한다.

## 7. 공지사항 & 배너 관리 기획

### 7-1. 앱 내 공지사항 CRUD

**목적:** 서비스 점검 공지, 이용약관 변경 안내, 이벤트 소개 등을 운영팀이 직접 작성하여 앱 내에 노출한다.

**필요한 신규 엔티티:**

현재 `Board` 엔티티(title, count, createdAt, updatedAt)가 이미 존재하나 실제 공지사항 기능으로 사용하기에는 필드가 부족하다. 확장 또는 별도 `Notice` 엔티티 신설을 권장한다.

```java
Notice {
    id: Long
    title: String           // 공지 제목
    content: TEXT           // 공지 본문 (마크다운 지원)
    noticeType: String      // GENERAL / MAINTENANCE / EVENT / POLICY
    isPublished: boolean    // 앱 노출 여부
    isPinned: boolean       // 상단 고정 여부
    publishedAt: LocalDateTime  // 노출 시작일
    expiresAt: LocalDateTime    // 노출 종료일 (null = 무기한)
    author: Member          // 작성자 Admin
    createdAt, updatedAt: LocalDateTime
}
```

**화면 구성:**

```
[공지사항 관리] /admin/notices
┌────────────────────────────────────────────────────────┐
│  [+ 새 공지 작성]                                      │
```

- 공지 삭제는 물리 삭제가 아닌 `isPublished=false`로 처리 (이력 보존)
- 고정 공지는 최대 3개 제한

**엣지 케이스:**
- 노출 종료일이 현재보다 이전으로 설정된 경우: "노출 종료일은 현재 이후여야 합니다" 오류

### 7-3. 타겟 푸시 알림

**목적:** 세그먼트별로 선별적 푸시 알림을 보내 이벤트 참여율 및 재방문율을 높인다.

**알림 세그먼트:**

| 세그먼트 코드 | 대상 | 기준 |
|--------------|------|------|
| `ALL` | 전체 활성 회원 | `status = ACTIVE` |
| `PHOTOGRAPHERS` | 작가 | `authority = WM or SA` + 사진 1장 이상 업로드 |
| `USERS` | 일반 사용자 | `authority = US` |
| `INACTIVE_7D` | 7일 이상 미접속 | `lastLoginAt < now() - 7days` |
| `NEW_USERS` | 신규 가입 30일 이내 | `createdAt > now() - 30days` |

**필요한 추가 인프라:** FCM(Firebase Cloud Messaging) 토큰을 Member 엔티티에 저장하는 컬럼 추가 필요.

**엣지 케이스:**
- 전체 발송 시 대상이 1,000명 이상인 경우: 비동기 배치 처리
- 푸시 발송 후 실패한 토큰 자동 정리 (FCM에서 invalid 토큰 반환 시 DB에서 제거)

**엣지 케이스:**
- 기존 문의가 있는 종류를 비활성화 시도 시: "이 종류로 접수된 문의 N건이 있습니다. 비활성화하시겠습니까?" 확인 팝업

### 8-3. 태그 정규화

## 9. 통계 & 리포팅 고도화

### 9-1. 작가 퍼포먼스 대시보드

**목적:** 개별 작가의 성과 지표를 측정하여 우수 작가를 발굴하고, 성과가 낮은 작가에게는 개선 방향을 제안한다.

**운영팀이 할 수 있는 것:**
- 작가별 사진 업로드 수, 좋아요/저장/공유 합산 조회
- 작가별 문의 수신 건수 및 읽음률 확인
- 상위 작가 랭킹(기간 선택) 조회
- 특정 작가 클릭 시 해당 작가의 회원 상세 페이지로 이동

**화면:**

```
[작가 퍼포먼스] /admin/stats/photographers
┌──────────────────────────────────────────────────────────────┐
│  기간: [이번달 ▼]                                           │
│                                                              │
│  랭킹 | 작가명       | 사진수 | 총좋아요 | 문의수 | 읽음률  │
│  1    | 이서연(@u2)  | 12장   | 1,234    | 15건   | 93%     │
│  2    | 김민준(@u1)  | 8장    | 987      | 12건   | 83%     │
│  3    | 웹관리자     | 5장    | 756      | 8건    | 100%    │
│                                                              │
│  하단: 선택 작가의 월별 퍼포먼스 추이 라인차트              │
└──────────────────────────────────────────────────────────────┘
```

**API 구현 포인트:**
- 집계 쿼리: `JOIN Member-Photo-Inquiry` + `GROUP BY member_id` + `SUM, COUNT`
- 기간 파라미터: `startDate`, `endDate` (기본: 이번달 1일~오늘)

### 9-2. 코호트 분석 (월별 신규 가입자 유지율)

**목적:** 특정 월에 가입한 사용자가 이후 달에도 활동을 유지하는 비율을 측정하여 리텐션 문제를 조기에 발견한다.

**측정 정의:**
- **가입 코호트:** 특정 월에 `createdAt`이 있는 회원 그룹
- **활성 기준:** 해당 월에 사진 업로드 or 문의 발송 1건 이상
- **표**: 코호트(행) × 가입 후 경과 개월(열) 매트릭스

**화면:**

```
[코호트 분석] /admin/stats/cohort
┌─────────────────────────────────────────────────────────┐
│  코호트      | M+0  | M+1  | M+2  | M+3  | M+4  | M+5  │
│  2026-01 (23명)| 100% | 61%  | 48%  | 39%  | 35%  | 31%  │
│  2026-02 (18명)| 100% | 56%  | 44%  | 38%  | 33%  | -    │
│  2026-03 (31명)| 100% | 65%  | 51%  | 42%  | -    | -    │
│                                                         │
│  색상: 높을수록 진한 녹색 (히트맵 형태)                 │
└─────────────────────────────────────────────────────────┘
```

**정책 제안:** 3개월 리텐션 목표를 40% 이상으로 설정. 40% 미만인 코호트 발생 시 해당 월의 가입자에게 리인게이지먼트 푸시 알림 발송을 트리거.

### 9-3. 수익화 지표 준비

**목적:** 향후 유료 플랜(예: 작가 프리미엄 구독, 문의 연결 수수료) 도입을 대비하여 핵심 지표를 미리 설계하고 데이터를 수집한다.

**KPI 설계 (수익화 전 선행 지표):**

| KPI | 측정 항목 | 목표 임계값 |
|-----|-----------|------------|
| MAU (Monthly Active Users) | 월 활성 회원 수 | - |
| 작가 활성율 | 이번달 사진 업로드한 작가 / 전체 작가 | 50% 이상 |
| 문의 전환율 | 문의 발송 / 사진 조회 | 트래킹 시작 |
| 작가 응답율 | 읽음 처리된 문의 / 전체 수신 문의 | 80% 이상 |
| 계약 성사율 | 계약 완료 / 전체 문의 | 목표 30% |
| NPS 대리 지표 | 저장수 / 좋아요수 비율 (깊은 관심도) | - |

**대시보드 레이아웃:**

```
[수익화 KPI] /admin/stats/kpi
- 핵심 지표 6개 카드 (현재값 + 전월 대비 증감)
- MAU 월별 추이 바차트 (최근 12개월)
- 문의 퍼널 전환율 시각화
```

### 9-4. 리포트 내보내기 (CSV/Excel)

**목적:** 정기 보고, 마케팅 분석, 외부 데이터 연동을 위해 Admin에서 조회한 데이터를 CSV/Excel로 다운로드할 수 있도록 한다.

**내보내기 지원 항목:**

| 데이터 | 형식 | 필터 조건 |
|--------|------|----------|
| 회원 목록 | CSV | 기간, 역할, 상태 |
| 사진 목록 | CSV | 기간, 무드, 작가 |
| 문의 목록 | CSV | 기간, 촬영종류, 읽음 여부 |
| 통계 요약 | Excel | 기간 |
| 작가 퍼포먼스 | Excel | 기간 |

**구현 방식:**
- 백엔드: Apache POI (Excel) + OpenCSV (CSV)
- 파일명 규칙: `happiness_members_20260619.csv`
- 개인정보 포함 다운로드는 WM 전용 권한

**엣지 케이스:**
- 조회 결과가 10,000건 이상인 경우: 비동기 생성 + 완료 시 다운로드 링크 제공
- 개인정보(이메일, 전화번호) 포함 여부를 내보내기 전 선택할 수 있도록 옵션 제공

## 10. 시스템 설정 고도화

### 10-1. 환경변수 실시간 수정

**목적:** 현재 시스템 설정 페이지는 읽기 전용이다. Rate Limit, 이메일 서버 설정을 재배포 없이 Admin에서 즉시 변경할 수 있도록 한다.

**운영팀이 할 수 있는 것:**
- Rate Limit 값 즉시 조정 (트래픽 급증 시 임시 완화)
- 이메일 서버 변경 (SMTP 이전 시)
- 변경 즉시 적용 확인

**필요한 DB 변경:**

```java
SystemConfig {
    id: Long
    configKey: String (unique)  // "rate_limit_capacity", "mail_host" 등
    configValue: String
    description: String         // 설명 (관리자 참고용)
    isEditable: boolean         // false인 경우 읽기 전용 표시
    updatedBy: Member
    updatedAt: LocalDateTime
}
```

**화면:**

```
[시스템 설정] /admin/system
┌──────────────────────────────────────────────────────┐
│  [이메일 설정]                   [수정]              │
│  MAIL_HOST:     smtp.gmail.com                       │
│  MAIL_USERNAME: admin@happiness.dev                  │
│  MAIL_PORT:     587                                  │

│  [Rate Limit]                    [수정]              │
│  최대 토큰:     100                                  │
│  리필 토큰:     10                                   │
│  리필 주기:     1초                                  │
│                                                      │
│  [수정 모달]                                         │
│  MAIL_HOST: [smtp.gmail.com      ]                   │
│  경고: 잘못된 설정은 메일 발송 오류를 유발합니다.   │
```

**정책:**
- 이메일 설정 변경 후 "테스트 메일 발송" 기능으로 즉시 검증
- Rate Limit 변경은 WM 전용
- 모든 설정 변경은 관리자 활동 로그에 기록 (이전값/변경값 모두 기록)

**엣지 케이스:**
- 잘못된 SMTP 호스트 입력 후 저장: 저장은 허용하되, 실제 발송 실패 시 시스템 알림
- Rate Limit을 0으로 설정: 0 이하 값 불허, 최솟값 1 강제

### 10-3. 관리자 활동 로그

**목적:** 어떤 Admin 계정이 언제 어떤 작업을 수행했는지 추적하여 내부 감사, 실수 복구, 보안 사고 조사에 활용한다.

**필요한 신규 엔티티:**

```java
AdminActivityLog {
    id: Long
    admin: Member           // 행위자 (WM/SA 계정)
    action: String          // "MEMBER_DELETE", "ROLE_CHANGE", "PHOTO_DELETE" 등
    targetType: String      // "MEMBER", "PHOTO", "INQUIRY", "SERIES", "SYSTEM"
    targetId: Long          // 대상 엔티티 ID
    beforeValue: TEXT       // 변경 전 값 (JSON)
    afterValue: TEXT        // 변경 후 값 (JSON)
    ipAddress: String
    userAgent: String
    createdAt: LocalDateTime
}
```

**기록 대상 액션:**

| 액션 코드 | 설명 |
|-----------|------|
| `MEMBER_ROLE_CHANGE` | 회원 역할 변경 |
| `MEMBER_STATUS_CHANGE` | 회원 상태 변경 (정지/해제) |
| `MEMBER_DELETE` | 회원 삭제 |
| `PHOTO_DELETE` | 사진 삭제 |
| `PHOTO_FEATURED_CHANGE` | 피처드 설정 변경 |
| `REPORT_PROCESS` | 신고 처리 |
| `SYSTEM_CONFIG_CHANGE` | 시스템 설정 변경 |
| `MAINTENANCE_TOGGLE` | 점검 모드 ON/OFF |
| `NOTICE_PUBLISH` | 공지사항 발행 |
| `BANNER_CHANGE` | 배너 변경 |

**화면:**

```
[관리자 활동 로그] /admin/system/audit-logs
┌──────────────────────────────────────────────────────────────┐

│  변경 전: {"status": "ACTIVE"}                              │
```

**정책:**
- 로그는 최소 1년 보관 (법적 요건 충족)

| 문의 성사율 통계 | `InquiryStatus` 확장 + 퍼널 차트 API + UI | 문의 퍼널 차트 표시, 성사율 수치 표시 |
| 작가 퍼포먼스 대시보드 | 집계 API + 랭킹 테이블 UI | 기간 선택 → 작가별 지표 표시 |
| shootType/colorMood DB화 | `ShootType`, `ColorMood` 엔티티 전환 + 관리 UI | 코드 배포 없이 신규 종류 추가 가능 확인 |
| 리포트 내보내기 | Apache POI/OpenCSV 연동 + 다운로드 API | CSV 다운로드 → 데이터 정합성 검증 |

**예상 소요:** 백엔드 3주 + 프론트 3주 (병렬 진행)

### Phase 3 — 장기 (8주 이상)

**목표:** 고도화 분석 및 수익화 준비

| 기능 | 작업 내용 | 완료 기준 |
|------|-----------|----------|
| 관리자 활동 로그 | `AdminActivityLog` 엔티티 + AOP 자동 기록 + 조회 UI | 모든 변경 액션 로그 자동 기록 확인 |
| 코호트 분석 | 코호트 집계 쿼리 + 히트맵 UI | 월별 리텐션 매트릭스 표시 |
| 수익화 KPI 대시보드 | KPI 집계 API + 지표 카드 UI | 6개 KPI 지표 실시간 표시 |
| 정렬 정책 관리 | `PolicySettings` 테이블 + 설정 UI + 앱 캐시 연동 | Admin 설정 변경 → 앱 피드 정렬 변경 확인 |
| 점검 모드 | 글로벌 인터셉터 + 설정 UI | 점검 ON → 앱 API 503 반환 확인 |
| 타겟 푸시 알림 | FCM 연동 + 세그먼트 발송 API + 발송 UI | 세그먼트별 푸시 발송 확인 |

**예상 소요:** 총 8~12주 (기능별 병렬 진행)
