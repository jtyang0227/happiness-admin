## 1. 서비스 포지셔닝

### 1-1. 앱과 Admin의 관계

```
happiness-app (사용자 앱)          happiness-admin (운영 도구)
─────────────────────────          ─────────────────────────────
작가 → 사진/시리즈 업로드     ←→   사진/시리즈 모더레이션, 피처드 설정
사용자 → 작가에게 문의 발송   ←→   문의 현황 모니터링, 스팸 탐지
사용자 → 부적절 콘텐츠 신고   ←→   신고 검토 및 처리
작가 → 인증 신청              ←→   인증 심사 및 승인/반려
사용자 → 공지사항 열람        ←→   공지사항/배너 CRUD
모든 사용자 → 앱 사용         ←→   통계 조회, 리포트 생성, 정책 설정
```

**Admin이 존재하는 이유:**
- 앱에서 발생하는 모든 콘텐츠와 트랜잭션을 **운영팀이 가시화**하고 **제어**할 수 있어야 한다
- 플랫폼 규정 위반, 스팸, 부적절 콘텐츠를 **신속하게 처리**하여 사용자 신뢰를 보호한다
- 추천/피처드 큐레이션을 통해 **플랫폼 콘텐츠 품질**을 유지한다
- 데이터 기반으로 **운영 정책을 수립**하고 효과를 측정한다

### 1-2. 운영자 역할 (WM / SA)

| 역할 | 명칭 | 주요 업무 | 접근 제한 |
|------|------|-----------|-----------|
| `WM` | 웹관리자 | 시스템 설정, 콘텐츠 정책, 관리자 계정 생성 | 모든 기능 |
| `SA` | 운영자 | 일상 콘텐츠 모더레이션, 문의 처리, 공지 작성 | 시스템 설정 제외 |

## 2. 앱 사용자 여정 × Admin 관리 매핑

### 2-1. 회원 (Member)

#### 앱에서 발생하는 일
```
[회원가입 / 로그인]
  → 이메일 or 소셜 로그인
  → 프로필 설정 (이름, 프로필명 @handle, 프로필 이미지)

[작가 활동]
  → 사진 업로드 (제목, 색채무드, 촬영종류 태그)
  → 시리즈 생성 (사진 묶음)
  → 작가 인증 신청 (포트폴리오, 경력 제출)

[일반 사용자 활동]
  → 사진 좋아요 / 저장 / 공유
  → 작가에게 문의 발송
  → 부적절 콘텐츠 신고
```

#### Admin에서 관리하는 것

| 앱 이벤트 | Admin 기능 | 우선순위 |
|-----------|------------|---------|
| 신규 가입 | 회원 목록 조회, 검색, 역할 관리 | ✅ 구현됨 |
| 규정 위반 | 회원 정지/해제, 탈퇴 처리 | P0 (즉시) |
| 작가 인증 신청 | 인증 심사 큐, 승인/반려 | P1 |
| 이상 행동 (스팸) | 이상 탐지, 벌크 제재 | P1 |
| 회원 민원 | 회원 상세 페이지 (활동 내역 통합 조회) | P0 (즉시) |

### 2-2. 사진 (Photo)

#### 앱 화면 구조
```
[메인 피드]
  정렬: 최신순 / 인기순 / 에디터 픽
  필터: 색채무드 (WARM/COOL/NEUTRAL/VIVID/DARK/SOFT)
           ↑ Admin에서 기본 정렬 정책 설정
           ↑ Admin에서 에디터 픽(피처드) 선정

[사진 상세]
  → 사진 이미지, 제목, 작가 정보
  → 좋아요 / 저장 / 공유 버튼
  → 작가에게 문의하기 CTA
  → 신고 버튼

[작가 프로필]
  → 업로드한 사진 그리드
  → 시리즈 목록
  → Verified 뱃지 (Admin이 부여)
```

#### Admin에서 관리하는 것

| 앱 화면 요소 | Admin 기능 | 우선순위 |
|-------------|------------|---------|
| 메인 피드 정렬 | 기본 정렬 정책 설정 (최신/인기/복합 가중치) | P3 |
| 에디터 픽 섹션 | 피처드 사진 10개 선정 + 순서 지정 | P2 |
| 색채무드 필터 | colorMood DB화 + Admin 관리 UI | P2 |
| 신고 버튼 | 신고 관리 시스템, 검토/처리 워크플로우 | P1 |
| Verified 뱃지 | 작가 인증 워크플로우 (심사/승인) | P1 |
| 사진 전체 목록 | 그리드/테이블 조회, 필터, 삭제 | ✅ 구현됨 |

### 2-3. 문의 (Inquiry)

#### 앱 화면 구조
```
[문의 발송 플로우]
  사용자 → 작가 프로필 또는 사진 상세 → [문의하기]
  → 촬영종류 선택 (웨딩/가족/프로필/스냅/바디프로필/커플)
           ↑ Admin에서 DB화 + 신규 종류 추가 관리
  → 희망 촬영일, 예산, 메시지 입력
  → 발송

[작가의 문의 수신함]
  → 미읽음/읽음 표시
  → 문의 내용 열람
  → 응답 (현재 앱 내 응답 기능 여부에 따라 다름)
```

#### Admin에서 관리하는 것

| 앱 기능 | Admin 기능 | 우선순위 |
|---------|------------|---------|
| 전체 문의 현황 | 문의 목록 (전체), 필터/읽음처리/삭제 | ✅ 구현됨 |
| 촬영종류 필터 | shootType DB화 + Admin 관리 UI | P2 |
| 스팸 문의 | 이상 문의 탐지 (24h 다수 발송, 동일 메시지) | P2 |
| 문의 성사율 | 퍼널 분석 (접수→읽음→응답→계약) | P2 |
| 수요-공급 분석 | 촬영종류별 문의 수 vs 작가 수 갭 분석 | P2 |

### 2-4. 시리즈 (Series)

#### 앱 화면 구조
```
[시리즈 목록]
  → 커버 이미지, 시리즈 제목, 작가명, 사진 수
  → Admin이 추천 시리즈 설정 가능

[시리즈 상세]
  → 시리즈에 포함된 사진 목록 (슬라이드/그리드)
  → 작가에게 문의하기 CTA
```

#### Admin에서 관리하는 것

| 앱 기능 | Admin 기능 | 우선순위 |
|---------|------------|---------|
| 시리즈 전체 목록 | 목록 조회 + 삭제 | ✅ 구현됨 |
| 추천 시리즈 | 피처드 시리즈 5개 선정 | P2 |

### 2-5. 공지 & 배너

#### 앱에서 필요한 것
```
[앱 내 공지사항 탭]
  → 운영팀이 작성한 공지 목록
  → 제목, 내용, 날짜 표시
  → 카테고리: 일반/점검/이벤트/정책

[메인 화면 배너 슬라이더]
  → 이미지 배너 (이벤트, 프로모션)
  → 클릭 시 특정 URL 이동
```

#### Admin에서 관리하는 것

| 앱 요소 | Admin 기능 | 우선순위 |
|---------|------------|---------|
| 공지사항 목록 | 공지 CRUD + 임시저장/발행/예약 | P1 |
| 메인 배너 | 배너 CRUD + 순서 관리 (드래그) | P1 |

## 3. Admin 화면 정의서

### 3-1. 현재 구현된 화면 (베이스라인)

| 화면 | URL | 핵심 기능 | 상태 |
|------|-----|-----------|------|
| 로그인 | `/login` | JWT 로그인 | ✅ |
| 대시보드 | `/` | KPI 카드, 최근 문의, 인기 사진, 바차트 | ✅ |

| 사진 관리 | `/photos` | 그리드, 필터, 삭제 | ✅ |
| 문의 관리 | `/inquiries` | 목록, 필터, 읽음처리, 삭제 | ✅ |
| 시리즈 관리 | `/series` | 목록, 삭제 | ✅ |
| 통계 | `/stats` | 꺾은선/파이/바 차트, TOP10 | ✅ |
| 시스템 | `/system` | 메일/Rate Limit 상태 (읽기전용) | ✅ |

## 4. 운영 시나리오 & 워크플로우

### 4-1. 시나리오: 부적절 콘텐츠 신고 처리

```
[트리거] 앱 사용자 "봄날의 추억" 사진 신고 (부적절 콘텐츠)
    ↓
[Admin Sidebar] 신고 관리 메뉴에 미처리 Badge 표시 (빨간 숫자)
    ↓
[SA 운영자] /reports 접속 → 미처리 신고 목록 확인
    ↓
[행의 [검토] 클릭] 신고 상세 SlideOver 오픈
    ↓
[사진 미리보기 확인] 실제 부적절 콘텐츠 여부 판단
    ↓
    ├── 부적절한 경우:
    │   [콘텐츠 삭제] 선택 → ConfirmDialog (danger)
    │   → 사진 DB 삭제 + 신고 status=ACTION_TAKEN
    │   → Toast "처리 완료" + 목록 갱신
    │
    └── 정상인 경우:
        [기각] 선택 → ConfirmDialog (warning)
        → 신고 status=DISMISSED
        → Toast "기각 처리 완료"
```

**총 소요 시간 목표**: 미처리 신고 1건당 30초 이내

### 4-2. 시나리오: 신규 작가 인증 심사

```
[트리거] 작가가 앱에서 "인증 신청" 제출
    ↓
[Admin] /verifications 신청 대기 목록 노출
    ↓
[SA] 포트폴리오 링크 클릭 → 외부에서 작품 확인

    │   → 앱에서 Verified 뱃지 즉시 표시

    └── 부적합 → [반려] 클릭 → 반려 사유 입력 필수
        → VerificationRequest.status = REJECTED

```

## 5. 기능 요구사항 (User Story)

### WM (웹관리자) 요구사항

```
US-WM-01: 시스템 점검 모드
  AS WM, 서비스 점검 시 앱 사용자 접근을 차단하고 싶다.
  GIVEN 점검 모드 설정 화면에서
  WHEN 점검 메시지와 종료 예정 시각을 입력하고 [활성화]를 클릭하면
  THEN 앱의 모든 API 요청에 503이 반환되고,
       앱 클라이언트에 점검 안내 화면이 표시된다.
  AND Admin (/api/admin/**)는 점검 중에도 접근 가능하다.

US-WM-02: 콘텐츠 정렬 정책 변경
  AS WM, 앱 피드의 기본 정렬 순서를 코드 배포 없이 변경하고 싶다.
  GIVEN 콘텐츠 정책 관리 화면에서
  WHEN 정렬 기준을 "인기순"으로 변경하고 [저장]을 클릭하면
  THEN DB에 정책이 저장되고, 앱은 5분 이내 정책을 반영한다.

US-WM-03: 관리자 활동 감사
  AS WM, SA 운영자가 수행한 모든 변경 이력을 조회하고 싶다.
  GIVEN 시스템 > 관리자 활동 로그 화면에서
  WHEN 특정 날짜 범위와 Admin 계정을 필터링하면
  THEN 해당 기간 동안의 역할 변경, 사진 삭제, 신고 처리 이력이
       변경 전/후 값과 함께 표시된다.
```

### SA (운영자) 요구사항

```
US-SA-01: 신고 처리
  AS SA, 앱에서 들어온 신고를 검토하고 처리하고 싶다.
  GIVEN 신고 관리 목록에서 미처리 신고를 클릭하면
  WHEN 신고 상세 SlideOver에서 대상 콘텐츠를 확인하고 [처리 완료]를 클릭하면
  THEN 선택한 조치(삭제/숨김/기각 등)가 즉시 적용되고,
       신고 상태가 "완료"로 변경되며, 처리 메모가 기록된다.

US-SA-02: 회원 정지 처리
  AS SA, 규정 위반 회원을 신속하게 정지하고 싶다.
  GIVEN 회원 목록에서 [정지] 버튼을 클릭하면
  WHEN 정지 사유와 기간을 입력하고 [정지 처리]를 클릭하면
  THEN 해당 회원이 즉시 앱 로그인에서 차단되고,
       회원 상태 Badge가 [정지]로 변경된다.

US-SA-03: 에디터 픽 설정
  AS SA, 주간 에디터 픽 사진을 선정하고 싶다.
  GIVEN 피처드 콘텐츠 관리 화면에서 [+ 사진 추가]를 클릭하면
  WHEN 사진 검색 Modal에서 사진을 선택하면
  THEN 에디터 픽 목록에 추가되고, 드래그로 순서를 조정할 수 있다.
  AND [저장] 클릭 시 앱 메인 에디터 픽 섹션이 즉시 반영된다.

US-SA-04: 공지사항 작성
  AS SA, 서비스 관련 공지를 앱 사용자에게 전달하고 싶다.
  GIVEN 공지사항 관리에서 [새 공지 작성]을 클릭하면
  WHEN 제목, 내용, 노출 기간을 입력하고 [발행하기]를 클릭하면
  THEN 공지가 즉시 앱 공지 API에서 조회되고,
       앱 사용자에게 표시된다.

US-SA-05: 문의 현황 모니터링
  AS SA, 플랫폼 전체 문의 현황을 파악하고 싶다.
  GIVEN 문의 관리 목록에서
  WHEN 특정 작가, 촬영종류, 읽음 여부로 필터링하면
  THEN 조건에 맞는 문의 목록이 표시되고,
       48시간 이상 미읽음 문의는 특별 표시된다.
```

### 핵심 원칙 (디자인 스펙 참고)

| 원칙 | 설명 |
